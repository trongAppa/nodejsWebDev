HTTP(HTTP, Hypertext Transfer Protocol) 프로토콜
- 웹에서 데이터를 주고받기 위해 사용되는 통신 프로토콜
- 클라이언트와 서버 간의 요청과 응답을 규정하여 웹 페이지의 전송과 표시를 가능

- 특징 : Stateless(무상태) / Connectionless(비연결성) / 평문(Plain text)으로 통신 / 요청(Request)과 응답(Response)로 구성
 - Stateless(무상태) : HTTP는 클라이언트와 서버 간의 연결을 유지하지 않으며, 각각의 요청과 응답은 독립적 / 이는 서버가 클라이언트의 이전 상태를 기억하지 않고 요청을 처리함을 의미
 - Connectionless(비연결성) : 클라이언트가 요청을 보내고 서버가 응답을 보낸 후 연결을 종료 / 각각의 요청은 독립적으로 처리
 - 기본적으로 평문(Plain text)으로 통신하며, 보안을 위해 HTTPS와 같은 암호화된 프로토콜을 사용할 수 있습니다.
 - 요청(Request)과 응답(Response)로 구성됩니다.

- 요청과 응답
 - 클라이언트는 HTTP 요청 메시지를 서버에 통신을 전송
 - 요청 메시지는 메서드, URI(Uniform Resource Identifier), 버전, 헤더 등으로 구성
 - 서버는 요청을 받고 처리한 후, HTTP 응답 메시지를 클라이언트에 반환 / 응답 메시지는 상태 코드, 헤더, 본문 등으로 구성

- HTTP 메서드
 - HTTP는 다양한 메서드를 제공하여 서버에 특정 동작을 요청
 - GET : 리소스의 조회를 요청
 - POST : 리소스를 생성하거나 수정하기 위해 데이터를 서버에 제출
 - PUT : 리소스를 서버에 업로드하거나 수정
 - DELETE : 리소스를 삭제

- 상태 코드
 - 서버는 응답 메시지의 상태 코드를 통해 요청의 성공 여부나 오류를 표시
 - 200 : 성공적인 요청을 나타냅니다.
 - 404 : 요청한 리소스를 찾을 수 없음을 나타냅니다.
 - 500 : 서버에서 오류가 발생했음을 나타냅니다.

- 웹 브라우저와 웹 서버 간의 통신을 단순하고 효율적으로 처리하기 위해 설계 / 이해하고 활용하는 것은 웹 개발에서 중요한 요소 중 하나

1) Stateless(무상태)
웹에서 데이터를 주고받을 때, 서버가 클라이언트의 이전 상태를 기억하지 않는것을 의미
예를 들어, 웹 페이지를 여러 번 요청하더라도 서버는 클라이언트의 이전 요청에 대한 정보를 기억하지 않음

이는 일상 생활에서 매번 새로운 요청을 하는 것과 비슷
예를 들어, 레스토랑에 가서 매번 주문을 할 때마다 이전 주문 내역이 기억되지 않는 것과 같음
상태 관리 없는 특징은 서버의 부담을 줄이고 확장성을 향상

2) 비연결성(Connectionless)
클라이언트와 서버 간의 연결을 유지하지 않는다는 의미
즉, 클라이언트가 요청을 보내고 서버가 응답을 보낸 후에는 연결이 자동으로 종료
예를 들어, 전화를 걸어서 상담을 받고 나면 통화가 종료되는 것과 비슷

매번 새로운 요청을 할 때마다 연결을 맺고 종료하는 것이기 때문에, 서버의 리소스를 효율적으로 관리할 수 있음
이렇게 HTTP 프로토콜의 상태 관리 없는 특징과 비연결성은 클라이언트와 서버 간의 통신을 단순하고 효율적으로 만듬

3) HTTP 통신 구조
HTTP 프로토콜을 통해 클라이언트는 서버에 요청을 보내고, 서버는 해당 요청을 수신하여 처리한 후 응답을 반환
HTTP 요청과 응답은 클라이언트와 서버 간의 통신에서 사용되는 메시지
각각의 메시지는 특정한 구조를 가지고 있으며, 요청과 응답의 내용을 정의하고 전달

HTTP 요청 구조
- 요청 라인(Request Line) : 요청 메서드, 요청 대상 URL, HTTP 프로토콜 버전으로 구성 / Ex) GET /index.html HTTP/1.1
- 헤더(Headers) : 헤더는 요청에 대한 추가 정보를 포함 / 헤더는 "이름: 값" 형식으로 구성되며, 한 요청에 여러 개의 헤더가 포함 / Ex) Host: www.example.com User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/91.0.4472.124 Safari/537.36 Content-Type: application/json
- 본문(Body) : 요청에 포함되는 데이터를 담고 있으며, 필요한 경우에만 사용 / POST 요청에서 데이터를 전송할 때 사용됩니다.

HTTP 응답 구조
- 상태 라인(Status Line) : 상태 코드, 상태 메시지, HTTP 프로토콜 버전으로 구성 / Ex) HTTP/1.1 200 OK
- 헤더(Headers) : 응답에 대한 추가 정보를 포함 / 헤더는 요청과 마찬가지로 "이름: 값" 형식으로 구성되며, 응답에 여러 개의 헤더가 포함 / Ex) Content-Type: text/html Content-Length: 1024
- 본문(Body): 응답에 포함되는 데이터를 담고 있음 / 주로 HTML, JSON, 이미지 등의 리소스가 포함

HTTP 요청과 응답은 이러한 구조를 가지고 있어 클라이언트와 서버 간의 통신에서 사용
요청은 클라이언트가 서버에게 어떤 동작을 요청하는 메시지이며, 응답은 서버가 클라이언트에게 동작의 결과를 알려주는 메시지
클라이언트는 서버에게 요청을 보내고, 서버는 해당 요청을 수신하여 처리한 후 응답을 반환
웹 페이지의 전송과 데이터 교환을 가능


4) HTTP 메서드
클라이언트가 서버에게 요청하는 동작을 지정하는 방법
각각의 메서드는 특정한 목적과 의도를 가지고 있으며, 서버는 이를 기반으로 요청을 처리

GET
- 서버로부터 정보를 요청할 때 사용
- 클라이언트가 서버에게 특정 리소스(웹 페이지, 이미지, API 데이터 등)를 요청하고, 서버는 해당 리소스를 응답으로 반환
- 주로 데이터를 조회할 때 사용되며, 요청 파라미터를 URL의 쿼리스트링에 포함시켜 전달

POST
- 서버로 데이터를 전송하여 새로운 리소스를 생성하거나 업데이트할 때 사용
- 클라이언트가 서버에게 데이터를 전송하고, 서버는 해당 데이터를 처리하여 새로운 리소스를 생성하거나 업데이트
- 주로 회원가입, 데이터 등록, 파일 업로드 등에 사용

PUT
- 서버에게 데이터를 전송하여 특정 리소스를 생성하거나 업데이트할 때 사용
- 클라이언트가 서버에게 데이터와 함께 특정 리소스의 위치(URI)를 지정하고, 서버는 해당 리소스를 생성하거나 업데이트
- POST와 유사하지만, PUT은 클라이언트가 명시적으로 리소스의 위치를 지정하는 점이 다름

DELETE
- 서버로부터 특정 리소스를 삭제할 때 사용
- 클라이언트가 서버에게 특정 리소스의 위치(URI)를 지정하고, 서버는 해당 리소스를 삭제
- 실수로 중요한 데이터를 삭제하는 것을 방지하기 위해 보안을 강화

HTTP 메서드는 위에서 설명한 것 외에도 다양한 메서드들이 존재
HEAD, OPTIONS, PATCH, TRACE 등이 있으며, 각각 특정한 목적과 동작을 수행
개발자는 메서드의 의미와 사용법을 이해하여 적절하게 선택하고 사용

5) HTTP 상태 코드
서버가 클라이언트에게 응답할 때 요청의 성공 여부나 오류 상황을 알려주는 코드
상태 코드는 세 자리 숫자로 이루어져 있으며, 각각의 코드는 특정한 의미

2xx (성공)
- 200 OK : 요청이 성공적으로 처리 / 서버가 클라이언트에게 요청한 정보를 정상적으로 반환
- 201 Created : 요청에 의해 새로운 리소스가 생성 / 주로 POST 요청 시 사용

3xx (리다이렉션)
- 301 Moved Permanently : 요청한 리소스가 새로운 위치로 영구적으로 이동 / 클라이언트는 새로운 위치로 요청을 재전송
- 302 Found : 요청한 리소스가 임시로 다른 위치에 있음 / 클라이언트는 임시로 다른 위치로 요청을 전송
- 304 Not Modified : 클라이언트의 캐시된 버전이 최신 상태임 / 서버는 클라이언트에게 실제 내용을 다시 보내지 않고, 클라이언트는 캐시된 내용을 사용

4xx (클라이언트 오류)
- 400 Bad Request : 잘못된 요청이 전송 / 서버는 요청을 이해할 수 없거나 유효하지 않은 요청이라고 판단
- 404 Not Found : 요청한 리소스를 찾을 수 없음 / 서버가 요청한 URI에 대한 리소스를 찾을 수 없는 경우 사용

5xx (서버 오류)
- 500 Internal Server Error : 서버에서 오류가 발생 / 서버는 요청을 처리하는 동안 예상치 못한 오류가 발생
- 503 Service Unavailable : 서버가 현재 요청을 처리할 수 없음 / 주로 서버가 과부하 상태이거나 유지 보수 중인 경우 사용

HTTP 상태 코드는 클라이언트에게 요청의 처리 상태를 알려주는 중요한 정보
개발자는 상태 코드를 확인하여 요청의 성공 여부나 오류 상황을 처리할 수 있도록 적절한 조치를 필요

6) HTTP와 관련된 보안과 성능 개선을 위한 기술
보안 측면에서, HTTP는 암호화되지 않은 텍스트 기반의 프로토콜이므로 데이터가 암호화되지 않고 전송
중간에 제3자가 데이터를 도청하거나 조작할 수 있는 보안 위협이 존재
HTTPS와 SSL/TLS와 같은 보안 기술을 도입하여 데이터의 암호화와 인증을 제공하여 보안성을 강화

성능 측면에서, HTTP는 매 요청마다 새로운 연결을 맺고, 중복된 데이터를 반복적으로 전송하는 단순한 요청-응답 모델
네트워크 대역폭의 낭비와 응답 시간의 증가가 발생
캐싱, 압축, HTTP/2와 같은 기술을 도입하여 데이터 전송의 효율성을 높이고 응답 시간을 단축시킬 필요가 있음

HTTP와 관련된 보안과 성능 개선 기술들이 필요하게 됨
보안을 강화하여 데이터의 기밀성과 무결성을 보호하고, 성능을 개선하여 빠른 응답과 효율적인 데이터 전송을 실현함으로써 웹 사용자들에게 안전하고 원활한 사용 경험을 제공함

HTTPS (HTTP Secure)
- HTTP의 보안 버전으로, 데이터의 암호화와 인증을 제공
- SSL(Secure Sockets Layer) 또는 TLS(Transport Layer Security) 프로토콜을 사용하여 클라이언트와 서버 간의 통신을 암호화함

중간에 제3자가 데이터를 도청하거나 조작하는 것을 방지하고, 안전한 통신을 보장

SSL/TLS
- 공개 키 암호화를 사용하여 클라이언트와 서버 간의 안전한 통신을 제공하는 암호화 프로토콜
- 클라이언트와 서버 간의 상호 인증, 데이터의 암호화 및 무결성 검증을 수행
- HTTPS는 SSL/TLS를 사용하여 HTTP 통신을 보호하는 것

캐싱
- 클라이언트가 리소스를 요청할 때 서버에서 해당 리소스를 전달하는 대신, 클라이언트나 중간에 위치한 캐시 서버에서 리소스를 반환하는 기술
- 캐싱을 통해 반복적인 요청에 대한 응답 속도를 향상

CDN(Content Delivery Network)
- 전 세계에 분산된 서버 네트워크를 통해 정적 컨텐츠를 캐싱하고 전송함으로써 전체적인 성능을 향상시키는 기술입니다.

압축
- 데이터 전송 시 데이터의 크기를 줄여서 네트워크 대역폭을 절약하고 응답 시간을 단축시키는 기술
- Gzip과 Deflate와 같은 압축 알고리즘이 사용되며, 서버는 리소스를 클라이언트에게 전송하기 전에 압축하여 전송
- 클라이언트는 해당 데이터를 압축 해제하여 사용

HTTP/2
- HTTP/1.1의 후속 버전으로, 성능 향상을 위해 설계된 프로토콜입니다.
- 다중화(multiplexing) 기능을 제공하여 한 커넥션에서 여러 개의 요청과 응답을 동시에 처리
- 또한 헤더 압축, 서버 푸시, 이진 프레이밍 등의 기능을 지원하여 네트워크 성능을 향상

HTTP와 관련된 보안과 성능 개선 기술들이 필요
보안을 강화하여 데이터의 기밀성과 무결성을 보호하고, 성능을 개선하여 빠른 응답과 효율적인 데이터 전송을 실현함으로써 웹 사용자들에게 안전하고 원활한 사용 경험을 제공
