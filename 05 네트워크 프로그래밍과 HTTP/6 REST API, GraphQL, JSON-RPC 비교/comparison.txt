REST API, GraphQL, JSON-RPC 비교
1. REST API
- Representational State Transfer의 약자로, 자원을 URI로 표현
- HTTP 메서드(GET, POST, PUT, DELETE 등)를 사용하여 해당 자원에 대한 CRUD(Create, Read, Update, Delete) 작업을 수행하는 방식
- 간단하고 직관적인 설계로, 캐싱, 스케일링, 코드의 가독성 등 여러 가지 장점
- 클라이언트가 필요한 데이터를 가져오기 위해 여러 번의 요청을 해야 할 수 있고, 과도한 데이터 오버페칭이 발생 가능성 있음

2. GraphQL
- 클라이언트가 필요한 데이터의 구조와 필드를 정확히 지정하여 요청하고, 서버는 해당 요청에 대한 데이터를 정확히 응답하는 쿼리 언어
- 클라이언트가 필요한 데이터를 한 번에 요청할 수 있으므로, 과도한 데이터 오버페칭과 언더페칭 문제를 해결
- 서버 개발 및 성능 최적화가 상대적으로 복잡할 수 있으며, 캐싱과 같은 기능을 직접 구현

3. JSON-RPC
- JSON을 기반으로한 경량의 원격 프로시저 호출 프로토콜
- 클라이언트가 JSON 형식의 요청 메시지를 생성하고, 서버는 해당 요청을 처리하고 결과를 반환하는 방식
- 단순하고 직관적인 구조로, 프로시저 호출에 특화되어 있습니다. 하지만 REST API나 GraphQL과는 달리 표준화된 방식이 아니며, 데이터의 구조를 명시적으로 정의하지 않기 때문에 문서화의 어려움이 있음

차이점을 고려하여 선택해야 할 API 스타일은 프로젝트의 특성과 요구사항에 따라 다를 수 있음
REST API : 간단하고 표준화된 방식으로 많이 사용
GraphQL : 클라이언트의 데이터 요구사항에 더욱 유연한 응답을 제공
JSON-RPC : 프로시저 호출에 특화되어 있어 단순한 API 구현에 유용
프로젝트의 특성과 요구사항을 고려하여 적절한 API 스타일을 선택
